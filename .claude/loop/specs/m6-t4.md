# M6-T4: 50 ページ目末尾ロック

## 目的
pageNumber = 50 で 30 行目末尾を越える入力を `onBeforeInput` で先読みキャンセルし、冊を 50 ページで閉じる。トースト等の通知は出さない（静けさ）。

## 変更対象
- ファイル:
  - `/home/cocotte/dev/diary/src/features/editor/EditorPage.tsx`
- 追加:
  - textarea の `onBeforeInput` ハンドラ
  - 判定: `current === PAGES_PER_VOLUME` の場合のみ、次の text を仮想計算 → `splitAtLine30` → overflow 非空なら `e.preventDefault()`

## 依存する既存コード
- T6.1 `splitAtLine30`
- `/home/cocotte/dev/diary/src/lib/constants.ts` `PAGES_PER_VOLUME = 50`, `LINES_PER_PAGE = 30`

## 実装概要（3-5行）
1. `handleBeforeInput(e: React.FormEvent<HTMLTextAreaElement> & { data?: string | null })` を定義。
2. `if (current !== PAGES_PER_VOLUME) return;`。
3. `const el = e.currentTarget; const { selectionStart, selectionEnd, value } = el; const inserted = (e as any).data ?? '';`（InputEvent の data プロパティ）。`nextValue = value.slice(0, selectionStart) + inserted + value.slice(selectionEnd);`
4. `const { overflow } = splitAtLine30(nextValue);` 。`if (overflow.length > 0) { e.preventDefault(); }`
5. 削除系 (inputType='deleteContentBackward' 等) は overflow が縮む方向なのでスルー（nextValue 計算と splitAtLine30 判定で自然に通る）。

## 受入条件チェックリスト
- [ ] 🔵 50 ページ目 30 行目末尾で改行しようとしても text が変わらない
- [ ] 🔵 49 ページ目までは通常通り入力・改行ができる
- [ ] 🔵 50 ページ目でも 30 行以内の入力は妨げられない
- [ ] 🔵 削除操作は妨げられない
- [ ] 🟡 ペーストで overflow を発生させる入力もキャンセルされる — 推測根拠: onBeforeInput はペーストにも発火

## Key Links（配線検証）
- `onBeforeInput` は EditorPage の `<textarea>` に直接配線される本番経路。
- T6.3 との関係: 本ロックが成立するページ (50) では T6.3 の自動遷移は発動しない（current === 50 でスキップ）。

## テスト方針
- ユニット (Vitest + Testing Library):
  - 50 ページ目に 30 行のテキストをあらかじめ入れておき、textarea の末尾にカーソルを置く
  - `fireEvent.beforeInput(textarea, { data: '\n' })` → preventDefault が発火、text が変わらない
  - 同じページで途中に文字挿入（overflow 発生しない）→ 通常通り入力される
  - 49 ページ目で同様の入力 → preventDefault 発火しない
- 目視: 50 ページ目を埋め、30 行目末尾で Enter を押すが何も起こらない（トースト無し）。

## 注意事項
- `onBeforeInput` は一部のモバイルブラウザで InputEvent の data が null のことがある。その場合は value の推定が難しいので、保険として `onChange` 側でも最終 value を `splitAtLine30` → overflow あれば値を元に戻す（prev に revert）実装を検討。
- IME 変換中は compositionupdate で文字確定前の beforeinput が走ることがある → composition 中は判定スキップ（`isComposingRef.current`）で安全側。
- 視覚フィードバックは plan.md の未解決事項 3 で「無し（静けさ原則）」を推奨採用。
