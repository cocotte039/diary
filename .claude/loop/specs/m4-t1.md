ルート `/` を本棚画面に割り当て、編集ルートを `/book/:volumeId/:pageNumber` に置き換える。既存 `/read/:id/:page` は新パスへリダイレクトする。

## 変更対象
- ファイル: `/home/cocotte/dev/diary/src/App.tsx`
- 関数/コンポーネント:
  - `App`（変更）: `<Routes>` の構成を差し替え
  - 新 `EditorPage` を T4.3 で追加するまでは仮インポート（plan.md の Wave 順に従い T4.3 完了後に本接続）
- 追加ルート:
  - `path="/"` → `BookshelfPage`
  - `path="/book/:volumeId/:pageNumber"` → `EditorPage`
  - `path="/read/:volumeId/:pageNumber"` → `<Navigate to="/book/:volumeId/:pageNumber" replace />`
  - `path="/write"` → `WritePage`（当面退避、M7 T7.5 で削除）
  - `path="/bookshelf"` → `<Navigate to="/" replace />`（任意。`/` が本棚になったため）

## 実装概要（3-5行）
1. `App.tsx` の import に新しい `EditorPage`（T4.3 で追加）を加える。
2. `<Route path="/">` の element を `BookshelfPage` に差し替える。
3. `<Route path="/book/:volumeId/:pageNumber">` を追加し `EditorPage` を割当。
4. `<Route path="/read/:volumeId/:pageNumber">` は `useParams` 経由で書き換えるリダイレクト用ラッパコンポーネントで処理するか、`<Route ... element={<ReadRedirect />} />` を作る（`:volumeId/:pageNumber` 込みで新 URL に replace navigate）。
5. `/write` ルートを別に割当て、WritePage を残す。

- `App.tsx` は `main.tsx` から呼び出される本番コードパス上にある。
- 追加する `ReadRedirect`（もしくはそれ相当の再配線）も `App.tsx` の `<Routes>` 内から直接レンダリングされる。テストのみの呼び出しとならないこと。

## テスト方針
- ユニット: Vitest + `MemoryRouter` で `/read/abc/3` を初期URLに指定し、`EditorPage` のテストマーカー（例: `data-testid="editor-page"`）がレンダリングされる、または `useLocation().pathname === '/book/abc/3'` となることを検証。
- ユニット: `/` で本棚のタイトル「本棚」テキストが出る。
- 目視: ブラウザで `/#/read/xxx/1` を貼り付け、`/#/book/xxx/1` に書き換わるか確認。

## 注意事項
- HashRouter 前提。`<Navigate>` ではパラメータが解決できないため、`ReadRedirect` 関数コンポーネントで `useParams()` → `navigate(replace:true)` する実装にする。
- M4 時点で WritePage ルートを残すのは回帰事故防止。M7 T7.5 で削除する。
- `/bookshelf` 古URL を踏む内部リンクは T4.6 で掃除する。
