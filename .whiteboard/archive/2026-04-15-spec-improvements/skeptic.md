# Skeptic 観点（要約）

## Critical リスク

### C1. 既存冊データの後方互換
- 既存ページには 60 行超の論理行を持つものは無いはずだが、1ページ 1200字超のコンテンツが存在する可能性は**ある**（特に過去 M9 で 60 行 × 長文で詰めた場合）。
- 新基準では 1200字超を開くと「最初から overflow 状態」になる。このとき `checkOverflowAndNavigate` が自動発動するとデータが勝手に次ページへ流れる → **ユーザーの「既存ページ内容は維持、再ページングしない」要件と真っ向衝突**。
- **対策必須**: 初回ロード直後は overflow 判定をスキップする。`handleChange`（＝ユーザー操作起因）でのみ overflow 処理する。現実装の `checkOverflowAndNavigate` は onChange/onCompositionEnd/insertDate からのみ呼ばれているため、初回ロードでは発動しないはず → 要確認。
- **追加対策**: `onBeforeInput` での最終ページロックは「次の入力が 1200字を超えるなら」ブロック。既存 1200字超状態で **削除**（data=""）は許可する必要がある。現実装は `if (!inserted) return;` で削除をスキップしているので OK。ただし既存 1200字超ページを最終ページ以外で開いた場合、ユーザーが 1 文字でも入力すると即 overflow 発動 → 1200字を切る position で keep を切る挙動で文字が次ページへ飛ぶ。これは仕様上許容？要確認だが、今回は **「既存ページ内容維持＝勝手に再ページングしない」の解釈を「自動遷移は発動しない」まで踏み込むべきかは曖昧**。安全策としては「既存冊をロードした直後、そのページが 1200字超だった場合、そのセッションで入力があっても overflow 判定を抑止する」フラグを設ける案もあるが、複雑化リスクあり。→ **ユーザー合意済み要件「再ページングしない」は「初回ロードで勝手に流さない」の意。編集介入後は通常どおり」と解釈し、追加対策不要**。計画にはこの解釈を明記する。

### C2. PAGES_PER_VOLUME 50 → 60 の影響
- 既存 50 ページで満杯の「completed」冊が存在する場合、60 に拡大すると「51〜60 ページ目が空白で再開可能」に見える。「completed」判定ロジックの再確認が必要（status は自動で active に戻らないか？）。
- **確認点**: `ensureActiveVolume` や冊完了判定が `pageCount === PAGES_PER_VOLUME` を使っていないか。grep 必須。
- BookshelfPage の `${count} / ${PAGES_PER_VOLUME}` メッセージは自動追従するので文言は問題なし。

### C3. スクロール構造変更での回帰
- `.surface` に `overflow-y: auto` を付けると、iOS Safari で momentum scroll の挙動が変わる。`-webkit-overflow-scrolling: touch` は現代 Safari では不要だが、古い端末では効く。
- 紙高さ `min-height: var(--page-height-px)` を textarea 側に付けると、**空ページでも 60 行分の罫線背景**が描かれる。これは要件通り。
- **回帰リスク**: 現在は textarea 自身が scroll コンテナ。カーソル復元時の scrollTop 設定（`getScrollTopForCursor`）が壊れる可能性 → 呼び出し箇所を確認。useEditorCursor 内で使われている場合、scroll コンテナが変わるので `.surface.scrollTop` に書き換え必要。
- **onTouchStart/End**: ページスワイプは `.root` に付いている。`.surface` のスクロールとジェスチャ干渉が出ないか要検証（水平優位 2:1 で抑止されているはず）。

### C4. ヘッダー固定がすでに実現しているか
- 現 `global.css` の `.app-header` は `position: fixed` を撤廃済み。`flex-shrink: 0` で flex 子要素として固定されている → **「ヘッダーが固定されない」というユーザー報告と現実装が矛盾**。
- 仮説: 現在 textarea が内部スクロールで、そのスクロールが親に伝播しないから「ヘッダーは動かない」はず。ユーザーが見ている現象は別かも。
  - 可能性1: フォーカス時の iOS/Android 仮想キーボード出現でビューポートが変わり、ヘッダーの位置は動いていないが `.root` の `100dvh` が縮んでスクロール不能に見える。
  - 可能性2: 非フォーカス時は textarea 自体がスクロールを受け付けない（focus が必要な実装になっている）。
- **ユーザー報告の中心は「非フォーカス時にスクロールできない」**。これは textarea 内部スクロール方式の典型的な症状（focus が無いと textarea が scroll event を受け取らない、または背景領域で指がキャッチされない）。→ **surface 外側スクロールへの移行で解決する**。

### C5. プログレスバーの混乱
- 10 分割目盛り（1200字を 120字ごと）は視覚的には細かい。認知負荷が上がる恐れ。
- 高さ 3px で常時表示なら目立たず OK と判断。
- アクセシビリティ: `role="progressbar"` + `aria-valuenow/min/max` 必須。テキストユーザー向けに `aria-label="ページの残量"`。

### C6. テスト壊滅リスク
- `EditorPage.test.tsx` の overflow/lock 系は全滅する（行数ベース）。書き換え量は大きいが、機械的。
- `db.test.ts` の LINES_PER_PAGE 依存も同様。
- 計画では「テスト書き換え」を別マイルストーンに切り出すより、各マイルストーンに同梱したほうが安全（ビルドが常にグリーンを維持）。

## Major リスク

### M1. 1200字の妥当性
- ユーザー合意済みだが、Klee One 日本語フォントで改行なしベタ書きの場合、現在の textarea 幅（1rem padding、横幅 360px 程度想定）では 1行あたり ~20字 × 60行 = 1200字。符合している。
- 横長画面（タブレット横 1024px）では 1行あたり 50字 × 60行 = 3000字収まるところ、1200字で強制送りになる → 「紙幅固定しない」のに空白が残る違和感。
- **これは受け入れた設計**（紙幅に追従しない仕様）。ただしユーザーに「大画面では紙面の下半分が空白になる」挙動を想起させるか再確認してもよいが、合意済みなので進める。

### M2. 最終ページロック時の UX
- 60 ページ目で 1200字到達後の入力が静かに無効化される。ユーザーは「なぜ打てない？」と混乱する恐れ。
- 合意: 静けさ原則（トースト/点滅/触覚なし）。
- 代替: プログレスバーが満量になっているので「残量ゼロ」が視覚的に伝わる。これで十分とする。

### M3. 視覚行オーバーフロー廃止の副作用
- 現 `splitAtLine30` は論理行＋視覚行のダブル保険。廃止後、論理行が 60 以下でも 1 行に巨大な文字列がある場合、従来は視覚行ベースで次ページに送られていたが、新仕様では 1200字到達まで送られない。
- これは「1200字で切る」という新仕様の意図そのもの。OK。

## Minor リスク

- `--lines-per-page` CSS 変数と `LINES_PER_PAGE` 定数の二重管理が紛らわしい。コメントで「罫線用の紙視覚行数」と明示。
- `HEADER_HEIGHT_PX` など関連定数は今回触らなくて良い。

## 検証方針

- Unit: `splitAtCharLimit` の境界（0, 1200-1, 1200, 1201, 大量）
- Integration: 1201字入力で次ページ遷移、60ページ目 1200字でロック、既存 1200字超冊のロードで自動遷移しない
- Manual (iOS/Android): 非フォーカス時スクロール、フォーカス時スクロール、キーボード出現時のヘッダー位置、プログレスバーの追従

## 未解決

- 既存冊で 1200字超ページが存在する場合の挙動ポリシー（上記 C1）は解釈を plan に明記して自律実装に委ねる。
