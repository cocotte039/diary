# Aesthete 分析 — 1ページ文字数上限撤廃

## 観点
視覚的美しさ・UX・一貫性・認知負荷・注意管理

## 1. 25 行ごと強調罫線の具体デザイン

### 1.1 デザイン目標

- ユーザーが「今何行目か」を無意識に把握できる「視覚的アンカー」
- 「静けさ」原則に反しない（目立ちすぎてはいけない）
- 書きながらは存在を忘れ、目を凝らせば気づける「地味な区切り」
- ページ送り判断のヒント（「25 行・50 行あたりでそろそろ切るか」）

### 1.2 色・濃度の提案

**現行罫線**: `--color-rule = rgba(255, 255, 255, 0.08)`
**強調罫線（提案）**: `rgba(255, 255, 255, 0.14)`

**根拠**:
- 0.08 → 0.14 は +0.06 の増加（1.75 倍）、視認可能な差だが静寂を保つ
- `--color-page-divider = rgba(255, 255, 255, 0.15)` と近い値にすることで「ページ区切り線」と視覚階層を揃える
- 0.20 まで上げると目立ちすぎてノート風の柔らかさが崩れる
- 0.10 では既存罫線との差が小さく効果がない

**推奨値レンジ**: 0.12 〜 0.16。Verify で主開発者が 3 段階（0.12 / 0.14 / 0.16）を実機比較して確定するのが理想だが、コスト考慮で **0.14 を初期値** として採用し、違和感があれば調整。

### 1.3 レイヤー重ね方

通常罫線の **Y 座標にピッタリ重ねる**（ズレを作らない）:

```css
background-image:
  /* 25 行ごと（強調） */
  repeating-linear-gradient(
    to bottom,
    transparent 0,
    transparent calc(25 * var(--line-height-px) - 1px),
    rgba(255, 255, 255, 0.14) calc(25 * var(--line-height-px) - 1px),
    rgba(255, 255, 255, 0.14) calc(25 * var(--line-height-px))
  ),
  /* 1 行ごと（通常） */
  repeating-linear-gradient(
    to bottom,
    transparent 0,
    transparent calc(var(--line-height-px) - 1px),
    var(--color-rule) calc(var(--line-height-px) - 1px),
    var(--color-rule) var(--line-height-px)
  );
background-size:
  100% calc(25 * var(--line-height-px)),
  100% var(--line-height-px);
```

**ポイント**:
- 25 行ごとの強調罫線を **上に重ねる**（CSS の multiple backgrounds は先頭が最前面）
- 強調罫線の y = 25 行目の既存罫線と同じ Y → 既存罫線の上にピッタリ重なって発色強化
- 線の太さ 1px は共通（太くすると浮く）
- `background-attachment: local` は親要素から継承

### 1.4 行数の根拠

- ユーザーは最終合意で **25 行** を指定（当初 20 行だったが 25 行に変更）
- 1 ページ 60 行 = 25 / 50 / 75 の位置に 2 本 (実際にはページ中に 25, 50 行目の 2 本)
- 1200 字 ÷ 60 行 ≈ 20 字/行 → 25 行は 500 字相当（進捗バー 40%〜50% 地点）
- 50 行目は進捗バー 80% 強地点 → 「そろそろ区切り時」の自然な目安

## 2. 進捗バー 100% 固定の心理的影響

### 2.1 現状の設計

```tsx
const progressPct = Math.min(
  100,
  Math.round((text.length / CHARS_PER_PAGE) * 100)
);
```

- 1200 字到達で 100%
- 1500 字でも 100%
- 塗り幅・色・tick はすべて変化なし

### 2.2 ユーザー視点でのシミュレーション

| 状況 | ユーザーの認知 | 感情 |
|---|---|---|
| 600 字書いた | 「半分」 | 順調 |
| 1200 字書いた | 「終わった」 | 一区切り感 |
| 1200 字超えた | **「ん？まだ書けるのか」** | 当初違和感 / 徐々に「自由度」に変換 |
| 2000 字書いた | 「今日は長く書いた」 | 達成感 / 充実感 |

### 2.3 懸念：100% 固定が「警告」に見える可能性

- 進捗バーが「満タン」で止まる = 一般 UI では「上限到達」のサイン
- 日記アプリでは「まだ書ける」のに「これ以上書くとどうかなる」と誤解する可能性

### 2.4 対策の選択肢

**案 A（推奨）**: **現状維持**
- 100% 固定でも、アプリ全体の「静けさ」原則から警告色やアニメーションは出さない
- ユーザーは数回試すうちに「100% 後も自由に書ける」と学習する
- 25 行ごと強調罫線が「進捗の別軸インジケータ」として機能する

**案 B**: 100% 到達後は「サブ進捗」を別スタイルで追加表示
- 例: 1200 字 → 100% / 1500 字 → 125% 相当の位置に薄い波線を追加
- → **採用しない**。静けさ原則に反する、実装コストも高い

**案 C**: 100% 到達時にバー全体の opacity を 0.5 → 0.3 に下げる（「もう気にしなくていい」の暗黙表現）
- 塗りの opacity が現状 0.5 → 100% 以後は 0.3 へ 120ms transition で下げる
- → **見送り**。現サイクルのスコープ外、Verify 後に別提案する選択肢として記録

**結論**: **案 A（現状維持）** を採用。

### 2.5 アクセシビリティ考慮

- `aria-valuenow={progressPct}` は 100 固定になる → スクリーンリーダーで「100%」と読まれ続ける
- `aria-valuenow` に 100 超の値を入れると aria-valuemax=100 と矛盾 → 現状の clamp は正しい
- `aria-valuetext` でより詳細な状態を表現する案もあるが、静けさ原則と合わない → 不要

## 3. ページ送り手動化による記述リズムの変化

### 3.1 旧 UX の問題点（再確認）

- スマホで端末幅が狭い → 1 行の視覚文字数が少ない → 60 行使い切る前に自動送りが走る → **最終行が中途半端な位置で切れる**
- 画面幅で「物理的ページ感」が変わる → 統一体験を壊す
- ユーザーは「この次の 1 文字で勝手にページが変わる」という不安を抱えながら書く

### 3.2 新 UX の期待される変化

- **書き手が区切りを決める** → 書き終えたあと、または思考が切れたところで手動送り
- **物理ノートの「ページを繰る」動作**に近づく → iOS ジェスチャー（スワイプ）との親和性
- **書いている最中は「ページ」を意識しない** → 没入感の向上
- **視覚アンカー（25 行強調罫線）**で「書きすぎ警告」なく自然に区切りを意識

### 3.3 リズムへの影響

| 旧 | 新 |
|---|---|
| 1200 字で強制ブレーク | 任意でブレーク |
| 書きながらページ送り意識 | 書き終えてから送る |
| スワイプは補助 | スワイプが主要動線 |
| 長文書きづらい | 長文も自由 |

### 3.4 情報構造の観点

- 1 ページ = 1 トピック の暗黙の単位は維持される（ユーザーが手動で区切るため）
- 逆に「1 トピックを 2 ページに渡る」ことも可能になる → 柔軟性向上
- 日付スタンプは手動挿入（既存）なので、複数日を 1 ページにまとめることも可能（設計上の自由度）

### 3.5 推奨アクション

- **説明文・ヘルプは追加しない**（静けさ原則）
- ユーザーは試行錯誤で新 UX を学習する
- 進捗バー・25 行罫線・手動ページボタンの既存要素の組み合わせで「書きすぎても大丈夫」のサインを暗黙伝達

## 4. 一貫性チェック

### 4.1 既存デザイントークンとの整合

- `--color-rule = rgba(255, 255, 255, 0.08)` との対比として 0.14 は許容範囲
- 将来トークン化する場合は `--color-rule-emphasis` のような命名案（本サイクルでは直指定で OK、アーカイブの M4 判断と同方針）

### 4.2 他画面との一貫性

- BookshelfPage / SettingsPage では罫線を描かない → 影響なし
- EditorPage のみ（編集画面専用の視覚アンカー）

### 4.3 ダークテーマ / 将来のライトテーマ

- 現状ダークテーマのみ、`rgba(255, 255, 255, ...)` 直指定
- ライトテーマを追加する場合はトークン化が必要になるが、本サイクル外

## 5. 認知負荷の変化

| 認知負荷要素 | 旧 | 新 |
|---|---|---|
| 自動遷移の予測 | 中（「あと何文字でページが変わる？」） | なし（自動遷移がない） |
| preventDefault される瞬間 | 中（「入力が効かない」混乱） | なし（ロックがない） |
| ページ送りの決断 | なし（自動） | 低〜中（自分で判断） |
| 書く位置の把握 | 中（行が途中で切れる不安） | 低（25 行ごと罫線＋進捗バー） |

**合計**: 認知負荷は **減る** と評価。「予測・混乱・不安」が消え「穏やかな判断」が残る。

## 6. Verify で確認すべき視覚項目

1. **25 行強調罫線の濃度**（0.14 が適切か、0.12 / 0.16 と比較したいか）
2. **強調罫線と通常罫線の Y 座標ピタリ一致**（1 ピクセルのズレがないか）
3. **iOS Safari / Android Chrome の文字レンダリング差**（罫線濃度の見え方が違う可能性）
4. **進捗バー 100% 固定の違和感**（数日使って不安が残るか、慣れるか）
5. **1200 字超のページのスクロール時に罫線が追従するか**（`background-attachment: local` の効果）
6. **手動ページ送り操作の記述リズムへの影響**（スワイプ / ボタン / PageUp/PageDown のどれを主用するか）

## 7. 将来課題（本サイクル外、記録のみ）

- **進捗バー 100% 到達時の opacity 微変化**（案 C）: ユーザーが「書きすぎ」を感じるなら検討
- **25 行強調罫線の濃度トークン化**: ライトテーマ追加時に必要
- **スタンプ挿入の自動改ページオプション**: 日付ごとに改ページしたいユーザー向け（設定項目として追加？）
- **冊終わりの視覚演出強化**: 60 ページ目到達時のトーン変化など
